# LimeStudio 技術ガイド：アーキテクチャと実装

このガイドでは、LimeStudio の「知覚 vs 現実」アーキテクチャと DirtyData カーネルの技術的な実装について深く掘り下げます。

> [!NOTE]
> LimeStudio の思想に初めて触れる方は、まず [メンタルモデル・ガイド](file:///Users/yasuno/projects/LimeStudio/docs/jp/MENTAL_MODELS.md) を読み、アーキテクチャ設計の背景にある概念を理解することをお勧めします。

---

## 1. 実行境界コントローラー (Execution Boundary Controller)

統合の核心は **Transaction Layer** であり、UI とオーディオエンジンの間の厳格なガードとして機能します。

### 1.1 意図 (Intent) vs パッチ (Patch)
- **意図 (Intent)**: ユーザーの欲求の表現（例：`Intent::TweakParam { ui_index, value }`）。UI によって生成されます。
- **パッチ (Patch)**: カーネルへの構造的な変更（例：`Operation::ModifyNode { id, delta }`）。Transaction Layer によって Intent からコンパイルされます。

### 1.2 ID アイソレーション (ViewCache)
「ゾンビ ID」を防止し UI の安定性を確保するため、2 層の ID システムを使用しています。
- **UI 空間**: `UiIndex` (u64)。GPU インスタンスバッファに最適化された、再利用可能な安定したインデックス。
- **カーネル空間**: `StableId` (ULID)。グラフの永続化と履歴管理に使用される、グローバルに一意な識別子。
- **BiMap**: `ViewCache` がこれら 2 つの空間のマッピングを維持します。

## 2. JIT 実行プラン

**DirtyData JIT** は、グラフをフラットな命令シーケンスに変換します。

### 2.1 実行ループ
エンジンは `DspOp` 命令を線形ループで実行します。
- **レジスタ**: オーディオバッファと状態用に事前割り当てられたメモリ。
- **アトミック操作**: `Add`, `Mul`, `Sin`, `Tanh`, `Accumulate` など。
- **決定論的乱数 (Deterministic PRNG)**: エンジン内でのランダム性は、`DeterministicRng` (Xoshiro256++) を介して制御されます。これにより、同一のシードからビット単位で再現可能なオーディオ出力が保証されます。
- **ノード統合**: レガシーまたは特殊なノードは `CallNode` 命令経由で呼び出され、安定した実行プランを維持しつつ複雑な DSP ロジックを可能にします。

### 2.2 待機のない通信 (Wait-Free Communication)
UI（知覚）スレッドとオーディオ（現実）スレッド間のすべての通信には、`rtrb` 待機なし SPSC キューを使用します。これにより、UI イベントやメモリ割り当てによってオーディオスレッドがブロックされることがなくなります。

- **Command Queue**: UI からエンジンへの単方向キュー。パッチ更新やパラメータ変更（Intent）を運びます。
- **Telemetry Bridge**: エンジンから UI への単方向高速イベントストリーム。CPU 負荷、クリッピング、NaN 検出、PRNG 状態の変化などをリアルタイムで UI に報告します。

## 3. SDK とプラグイン・コントラクト（カーネルの法）

`dirtydata-sdk` は、プラグイン ABI の真実のソースです。

### 3.1 `plugin!` マクロ
```rust
plugin! {
    name: "my_filter",
    params: {
        cutoff: { min: 20.0, max: 20000.0, default: 1000.0, label: "Cutoff" }
    },
    process: |ctx, inputs, outputs, self_p| {
        // 安全でブロックベースの DSP ロジック
    }
}
```
このマクロは、必要な ABI ボイラープレート、マニフェスト JSON、静的なインスタンス管理を生成し、開発者が「知覚に依存しない」純粋な DSP コードを書けるようにします。

## 5. フォレンジック監査（三種の神器）

LimeStudio は、オーディオソフトウェアの完全性と信頼性を確保するためのフォレンジック・ツール群を提供します。

### 5.1 LimeLint (静的な裁判官)
`cargo lime lint` は、プラグインのソースコードと UI 定義に対して深い静的解析を行います。
- **視覚法**: 非準拠の HIG v3.0 属性を拒否します。
- **アーキテクチャの完全性**: パラメータ権限をバイパスするローカルな状態変更を検出します。
- **リアルタイム安全性**: DSP パス内での非リアルタイム安全な呼び出し（アロケーション、ロック、I/O）を特定します。

### 5.2 Lime Doctor (生存診断)
`cargo lime doctor` は、既知のホスト固有の異常に対してプラグインを監査します。
- **ホストプロファイル**: Logic Pro, FL Studio, Ableton のライフサイクルをシミュレートします。
- **アンチパターン検出**: 状態の重複や手動レイアウト計算などの「JUCE 臭（JUCE-smell）」をフラグ立てします。

### 5.3 Lime Verify & Testify (フォレンジックな証跡)
- **`verify`**: `.lime` プリセットが報告されたハッシュと一致することを暗号的に証明します。
- **`testify`**: プリセットをその正確な Git コミットと DSP ソースコードに結びつけるフォレンジック・レポートを生成します。

## 6. 出荷の儀（The Release Ritual）

`cargo lime release` コマンドは、製品版デリバリーのための義務的な手順です。
1.  **厳格な監査**: `lint --strict` と `doctor` を実行します。
2.  **プリセット検証**: `verify` 経由ですべての同梱ファクトリープリセットを検証します。
3.  **リリースビルド**: 最適化された最終バイナリをコンパイルします。
4.  **封印**: 環境と整合性ハッシュを含む `SHIPMENT_MANIFEST.txt` を生成します。

---

## 4. 運用上の安全性（敵対的検証）

安全性はすべての命令に統合されています。
- **AssertRange**: コンパイラによって注入され、実行時に範囲外のシグナルを検出します。
- **AuditReport**: カーネルはすべての違反に対して診断レコードを生成し、それが UI の安全性モニターに反映されます。

## 7. LimeSurface: 宣言的 UI フレームワーク

LimeSurface は、GPU 最適化された SDF 描画エンジンを、Flutter のような宣言的なプログラミングモデルで操作するためのフレームワークです。

### 7.1 Everything is a Widget
UI の最小単位は `Widget` トレイトを実装する構造体です。すべてのウィジェットは `build()` メソッドを持ち、その瞬間の状態に基づいた UI 構造（`WidgetIR`）を返します。

- **Composition**: 複雑な UI は、小さなウィジェットを組み合わせることで構築されます。
- **Stateless**: プラグイン側での UI 定義は、現在のパラメータ状態から UI 構造への純粋な投影（Projection）として機能します。

### 7.2 物理駆動のインタラクション
LimeSurface のすべてのアニメーションは、線形補間ではなく **Critically Damped Spring（臨界減衰スプリング）** モデルに基づいています。
- **Higher Fidelity**: ユーザーの操作やシグナルの変化に対して、物理的に自然で「重み」のある反応を返します。
- **HIG 準拠**: すべてのモーションは 8px グリッドに整合し、不自然な中間状態を排除します。

### 7.3 Custom Widgets（脱出口）
標準の `Knob` や `Slider` では不十分な場合、`Custom` ウィジェットを使用して低レベルの SDF プリミティブを直接操作できます。これにより、エンジン本体を改造することなく、完全に独自のビジュアライザーや特殊な操作子をプラグイン側で実装可能です。
