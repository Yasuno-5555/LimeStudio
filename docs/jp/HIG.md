# Lime HIG v3.0 — 助言ではなく、法として

**プロジェクト:** Lime Studio  
**バージョン:** 3.0 (法執行版)  
**対象プラットフォーム:** CLAP / VST3 / AU / Rust (wgpu / nih-plug)  
**デザイン原則:** マット、精密、決定論、強制された一貫性

## 1. 基本理念

Lime Studio の UI は装飾ではありません。

ブランディングではありません。
「喜び」をエンジニアリングするものでもありません。
「モダンな感じ」を出すためのものでもありません。

それは **精密機器** です。

インターフェースは、注意力を維持し、因果関係をさらし、曖昧さを排除するために存在します。

すべてのウィジェットは以下の問いに答えなければなりません：
- 何が変わったのか？
- なぜ変わったのか？
- ユーザーはそれを信頼できるか？

答えが不明確な場合、そのウィジェットは無効（Invalid）です。

---

## 2. 絶対的な視覚ルール

これらのルールは義務的です。違反は「好み」の問題ではなく、デザインエラーです。

### 2.1 マット ＆ ソリッド (Matte & Solid)
**禁止事項:**
- グロー (Glow)
- ブルーム (Bloom)
- 影 (Shadow)
- 光沢のあるグラデーション
- 偽の反射
- ニューモーフィズム (Neumorphism)
- 装飾的なぼかし
- 装飾的なアニメーション粒子

**必須事項:**
- フラットな表面
- ソリッドな塗りつぶし
- 幾何学主導の強調
- 明示的な境界線

視覚的な階層は以下によって構築されなければなりません：
- スペーシング
- ストロークの幅
- 形状の変化
- スケールの差

劇場のようなライティングによる演出は決して行わないでください。私たちはカジノのインターフェースではなく、楽器（インストゥルメント）を構築しているからです。

### 2.2 色よりも形状 (Shape Over Color)
状態はまず幾何学（形状）によって伝えられなければなりません。色は二次的なものです。

**優先順位:**
形状 (Shape) > 動き (Motion) > コントラスト (Contrast) > 色 (Color)

**例:**
- **良:** バイパス = スイッチの位置が変わる。アクティブなモジュレーション = リングの厚みが変わる。クリッピング = 境界線が変形する。
- **悪:** 「赤く光る」（それはデザインではなく、ただのパニックです。）

### 2.3 色空間 (Oklab)
すべての補間は Oklab で行われなければなりません。sRGB での補間は禁止されています。

**必須:**
- スペクトラム・アナライザー
- モジュレーション・ヒートマップ
- グラデーション
- メーター
- ダイナミック・オーバーレイ

なぜなら、濁ったグラデーションは文明を崩壊させるからです。

## 3. タイミングのルール

### 3.1 アニメーション標準
すべての UI 遷移は、デフォルトで **物理演算 (Critically Damped Spring)** を使用しなければなりません。

**標準定数:**
- **Standard:** 剛性 300.0, 減衰 35.0 (最も一般的な遷移)
- **Fast:** 剛性 600.0, 減衰 50.0 (ボタンの反応など)
- **Slow:** 剛性 150.0, 減衰 25.0 (背景の変化など)

線形補間 (Linear) は、意図的な「機械的・デジタル的な不自然さ」を演出する場合を除き、原則禁止されています。

### 3.2 物理挙動 (Physics Motion)
物理挙動は、触覚的な意味が存在する場所にのみ存在します。ボタンに情緒的な深みは必要ありませんが、すべてのパラメータ遷移は物理的に整合していなければなりません。

## 4. タイポグラフィ
- **フォント:** JetBrains Mono (これのみ。例外なし。)

| 役割 | サイズ |
| :--- | :--- |
| 主要な値 | 14px |
| 二次的なラベル | 11px |
| 警告状態 | 14px |
| コードビュー | 13px |

**ルール:**
- すべての数値は等幅で整列していなければなりません。
- ラベルは値と競合してはなりません。
- パラメータラベルには大文字を推奨します。

ユーザーが読むのは「バイブス」ではなく「数値」です。

## 5. フォーカス・システム

これは必須です。これがないと、すべてが苦痛に変わります。

### 5.1 フォーカス可能なウィジェット
以下のみがフォーカスを受け取ることができます：
- NumberBox
- TextInput
- ListView
- DropDown
- InteractiveEnvelope
- CodeView (選択モード)

ノブはキーボードフォーカスを受け取りません。人間はすでに十分に混乱しているからです。

### 5.2 フォーカスリング
**必須スタイル:**
- 1px のアウトライン
- 落ち着いた Lime のアクセントカラー
- グローなし
- パルスなし
- アニメーションなし

フォーカスは目に見える必要がありますが、ドラマチックである必要はありません。

### 5.3 キーボード・ルール
- **Tab:** 次のフォーカス可能なウィジェットへ移動
- **Shift + Tab:** 前のウィジェットへ移動
- **Enter:** 確定 (Commit)
- **Escape:** キャンセルし、直前の確定値を復元
- **ダブルクリック:** パラメータをデフォルト値にリセット

これは普遍的でなければなりません。「プラグインの気分」に左右されてはいけません。

---

## 6. パラメータ権限 (Parameter Authority)

最も重要なルールです。注意深く読んでください。

**真実モデル:**
- パラメータ = 真実 (Truth)
- ウィジェット = 投影 (Projection)

ウィジェットはパラメータの状態を所有してはなりません。決して。ローカルな真実も、現実のふりをした楽観的な嘘も、隠されたキャッシュも認められません。

UI は状態を反映するものであり、捏造するものではありません。

**必須のフロー:**
`UI → パラメータ層 → エンジン → 投影 → UI`

決して `UI → DSP 直接変更` としないでください。（それは狂気への道です。）

---

## 7. インタラクションの閾値

一貫性は信頼に繋がります。

### ノブ (Knob)
- **精密ドラッグ:** Shift
- **精密倍率:** 0.1x
- **ダブルクリック:** リセット
- **デッドゾーン:** 3 論理ピクセル (px)
- **ホイールステップ:** 正規化された 1 ステップ

### スライダー (Slider)
- **スナップ閾値:** 4 論理ピクセル (px)
- **精密モード:** Shift
- **リセット:** ダブルクリック

### 数値ボックス (NumberBox)
- **ドラッグ:** 垂直方向
- **確定:** Enter
- **キャンセル:** Escape
- **無効な入力:** 静かに拒否

タイポ（入力ミス）でユーザーを罰しないでください。彼らはすでに苦しんでいます。

---

## 8. 密度（密度）のルール

スペーシングは装飾ではなく、アーキテクチャです。

**標準単位:**
- **基本単位:** 8px

すべてのスペーシングは 8 の倍数でなければなりません。勝手な 13px などというナンセンスは認められません。

| 要素 | 最小値 |
| :--- | :--- |
| ウィジェットのヒットエリア | 32px |
| パネルのパディング | 16px |
| ノブからラベルまでの距離 | 8px |
| セクション間のギャップ | 24px |

## 9. エラーのセマンティクス

警告は一貫していなければなりません。パニックは UX ではありません。

**状態:**
- **警告 (Warning):** 落ち着いたアンバー色（クリッピング、オーバーサンプリングの警告、不安定なフィードバックのリスク）
- **致命的 (Critical):** まず形状の変形（NaN、無効なグラフ、DSP の失敗）

赤色だけでは不十分です。色覚特性を持つユーザーも、機能的なコンプレッサーを使う権利があります。

---

## 10. コマンドモデル

ボタンは危険です。慎重に使用してください。

**カテゴリ:**
- **アクション:** 即時実行（例：プリセット保存）
- **ミューテーション:** 永続的な状態変化（例：バイパスの切り替え）
- **可逆アクション:** Undo/Redo と統合されていなければならない（例：ルーティングの削除）

これらを混ぜてはいけません。特に「とりあえず動かしたかっただけ」という理由で混ぜることは許されません。

---

## 11. ウィジェットの制限

**デフォルトで禁止:**
- アニメーションするグラデーション
- 浮遊する装飾的なアイコン
- ウィジェットごとの独自の独立したアニメーションタイミング
- 任意のアクセントカラー
- システム半径を超える角丸
- 隠されたモーダル状態

説明が必要なウィジェットは、失敗したウィジェットです。

---

## 12. 執行 (Enforcement)

このドキュメントは提案ではなく、**「実行可能な法」** です。

**必須ツール:** `cargo lime-lint`

以下を検証しなければなりません：
- 不適切なスペーシング
- アニメーションの違反
- 色の違反
- フォーカス処理の欠落
- 誤ったパラメータ所有権
- 非 Oklab の補間
- 不正なウィジェット状態

CI は違反時に失敗しなければなりません。

コンパイラによって強制された美もまた美です。おそらく、より優れた美でしょう。
