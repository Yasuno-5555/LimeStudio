# LimeStudio Advanced Internals: The Reality Engine

このドキュメントは、Rust の型システム、メモリモデル、およびリアルタイム・システムに精通した上級開発者を対象に、LimeStudio の内部実装とアーキテクチャの核心を解説します。

---

## 1. 知覚と現実の調停 (Reconciliation Model)

LimeStudio の最も重要な抽象化は、**Transaction Layer** による「楽観的更新」と「確定」の分離です。

### 1.1 Intent のコンパイル
UI から発行された `Intent` は、Transaction Layer によって `ULID` ベースの `trace_id` を付与され、`PatchSet` へとコンパイルされます。
- **楽観的反映**: `AddNode` や `SetParameter` は、カーネルからの応答を待たずに UI（Perception）に即時反映されます。
- **副作用の抑制**: `RemoveNode` など破壊的な操作は、カーネルの確定を待ってから反映されます。

### 1.2 ID アイソレーション
- **UI Space**: GPU インスタンスバッファの効率化のため、`u64` ベースの `UiIndex` を使用。
- **Kernel Space**: 全体の一貫性と履歴管理のため、`ULID` ベースの `StableId` を使用。
- `ViewCache` がこの双方向マッピングを保持し、エンジンの再起動やグラフの復元時にも UI のアイデンティティを維持します。

## 2. DirtyData Kernel: JIT & Execution Plan

エンジンの心臓部である `DirtyData` は、グラフ構造をそのまま実行するのではなく、線形な実行プランへとコンパイルします。

### 2.1 実行ループの解剖
`ExecutionPlan` は以下の要素で構成されます：
- **Registers**: オーディオバッファおよびノード間通信用の事前割り当て済みスタック。
- **DspOp**: 最小単位の DSP 命令列。
- **Node Integration**: 重い処理や外部ライブラリは `CallNode` を通じて呼び出されますが、レジスタの読み書き自体はコンパイラが管理します。

### 2.2 待機のない通信 (Wait-Free SPSC)
UI スレッドとオーディオスレッド間の通信には `rtrb`（Wait-free SPSC queue）を使用しています。
- **UI -> Engine**: `PatchEvent`（パラメータ変更、グラフ構造の変更）。
- **Engine -> UI**: `Snapshot`（現在の物理的状態、解析データ）。
これにより、オーディオレッドでのロックやアロケーションを完全に排除しています。

## 3. ポリフォニック・ボイス・マネジメント

`limestudio_core/src/engine.rs` で実装されているボイス管理は、高度な「ボイス・スティーリング」アルゴリズムを採用しています。

### 3.1 スコアリング・システム
各ボイスの「重要度」を以下の式（概念）で算出します：
`Score = StateImportance + LevelBonus - AgePenalty`
- アイドル状態のボイスは無限大（最も盗まれにくい）。
- リリース中のボイスや音量の小さいボイス、古いボイスは優先的にスティーリングの対象となります。

### 3.2 クリック防止の Stealing 状態
ボイスを盗む際、LimeStudio は即座に波形をリセットしません。
1. **Stealing 状態への遷移**: 対象ボイスに 220 サンプルのフェードアウトを開始。
2. **ゲート・オフ**: `gate` パラメータを 0.0 に設定。
3. **遷移完了**: フェードアウト完了後にレジスタをクリアし、新しいノートを開始。

## 4. `plugin!` マクロのメタプログラミング

`limestudio_macro` は、単なる定型文の生成器ではなく、**静的解析の注入器**として機能します。

- **静的インスペクション**: `dsp` ブロック内のコードを解析し、リアルタイム安全性に違反する関数呼び出し（`println!`, `Vec::push` 等）を検出します。
- **自動インストルメンテーション**: `CodeView` ウィジェットを UI に自動注入し、実行中の DSP コードをユーザーが「目視」できるようにします。
- **ABI ブリッジ**: `nih-plug` の複雑なライフサイクルを隠蔽し、`Params` 構造体の自動生成と `PatchEvent` へのマッピングを自動化します。

---

## 5. SDF ベースの UI ランタイム

`limestudio_surface` は、ビットマップ画像や伝統的なベクター描画を使用せず、**Signed Distance Fields (SDF)** を用いた `wgpu` 実装です。

- **解像度独立性**: 拡大・縮小してもエイリアシングが発生せず、常に完璧なエッジを保ちます。
- **低レイテンシ**: GPU のフラグメントシェーダーで図形を直接生成するため、CPU 負荷が極めて低く、オーディオ処理にリソースを割くことができます。
- **知覚的整合性**: カラー補間はすべて `Oklab` 空間で行われ、sRGB 特有の「色の濁り」を排除しています。

## 開発者への挑戦

LimeStudio の内部は、徹底的な **「不変性（Reality）」** と **「流動性（Perception）」** の戦いです。もしあなたが新しい `DspOp` を追加したり、レンダリングエンジンの SDF ロジックを最適化したいのであれば、それは「現実を書き換える」ことに他なりません。

ソースコードの `limestudio_core/src/torture.rs` にあるストレスリグを実行して、あなたの変更が過酷な環境下でも「信頼」を維持できるか確認してください。

## 6. 法を逃れる：LimeLint を外した開発 (The Outlaw Workflow)

LimeLint は「信頼」と「準拠」を保証しますが、研究目的や実験的な UI パラダイムを試す場合、上級開発者はこれらの制約を回避する必要があるかもしれません。

### 6.1 特定の法の無効化
プロジェクトルートの `lime-lint.toml` を修正することで、特定のチェッカーを沈黙させることができます。
- **UI の柔軟性**: `visual.forbidden_attributes = []` に設定することで、影やグローを許可します。
- **アーキテクチャの自由**: `architectural.enforce_parameter_authority = false` に設定することで、ウィジェット内のローカル状態を許可します。

### 6.2 「アウトロー」モード
完全な自由を得るために、`limestudio_plugin` 高レベル SDK を完全にバイパスすることも可能です。
1.  **カーネルの直接統合**: 標準的な `nih-plug` プロジェクト内で `dirtydata-runtime` を直接使用します。
2.  **手動の実行プラン**: `PlanCompiler` を使用して独自の `ExecutionPlan` を構築し、`process` ループ内で実行します。
3.  **UI の自由**: `Widget` IR の制約を受けずに、`limestudio_surface` を生の `wgpu` レンダラーとして使用します。

> [!CAUTION]
> LimeLint をバイパスすると、「確信（Confidence）」の保証が失われます。`cargo lime release` コマンドは、これらのビルドを出荷マニフェスト内で **「Unverified/Outlaw（未検証/アウトロー）」** としてフラグ立てし、「信頼第一」の配信プラットフォームでは拒否される可能性があります。
