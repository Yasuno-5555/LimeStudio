# LimeStudio メンタルモデル：インタラクションから現実まで

LimeStudio は、従来の UI やオーディオフレームワークから来ると直感に反するかもしれない、厳格なアーキテクチャ原則に基づいています。このガイドでは、LimeStudio を使いこなすために必要な 2 つの主要な「メンタルシフト」について説明します。

---

## 1. 知覚（Perception）vs 現実（Reality）：状態投影モデル

従来の UI プログラミングでは、ウィジェット（ノブなど）が通常その値を「所有」します。ノブを回すと、ローカルの状態が更新され、他のコンポーネントに変更を通知します。

LimeStudio において、**ウィジェットは幻影（Hallucination）です。** それらは何も所有していません。

### 1.1 「ローカルな真実の不在」原則
私たちは厳格な **Execution Boundary Controller（実行境界コントローラー）** パターンに従っています。アプリケーション全体において、真実のソースはただ 1 つ、**カーネル（Kernel）** です。

- **知覚 (Perception/UI)**: 私たちが状態であると「考えている」ものの反映。
- **現実 (Reality/Kernel)**: オーディオスレッドで実行されている、実際の、フォレンジックな状態。

### 1.2 コミュニケーション・ループ
直接的な変更（Mutation）の代わりに、すべては以下の強制的な 5 ステップのループに従います。

1.  **意図 (Intent)**: あなたがノブをドラッグします。UI は `Intent::TweakParam { id: "gain", value: 0.5 }` を発行します。
2.  **トランザクション**: コア（Transaction Layer）が Intent を受け取り、検証します（例：「0.5 は範囲内か？」）。
3.  **現実の更新**: カーネルが実際の DSP パラメータを更新します。ここが値が真に存在する **唯一の** 場所です。
4.  **スナップショット (真実)**: カーネルがその状態のスナップショットを UI に送り返します。
5.  **投影 (Projection)**: UI がスナップショットを受け取り、ノブの位置を 0.5 に更新します。

### 1.3 なぜこの「冗長性」が必要なのか
- **リアルタイム安全性**: UI スレッドとオーディオスレッドは物理的に分離されています。このループを使用することで、UI がオーディオスレッドをブロックしないこと、およびオーディオスレッドが UI でレースコンディションを引き起こさないことを保証します。
- **フォレンジックな監査可能性**: すべての変更が個別の `Intent` であるため、パラメータが *なぜ* 変更されたのか（ユーザー？LFO？オートメーション？）を正確に追跡できます。
- **楽観的 UI**: ループが遅く感じる場合、UI は状態を「予測」して表示できます（楽観的更新）。しかし、それは常にカーネルのスナップショット（究極の権威）によって修正されます。

> [!TIP]
> **ループに抗わないでください。** ウィジェットに `self.current_value` を追加しようとしている自分に気づいたら、止まってください。値は常に render 関数に渡される `Snapshot` から取得する必要があります。

## 2. Squeezer：知覚と現実の架け橋

この「知覚 vs 現実」のサイクルを、直感的に管理するための環境が **Squeezer IDE** です。

Squeezer は単なるエディタではなく、以下の役割を果たします：
- **可視化された因果関係**: ユーザーが操作した `Intent` がどのように DSP グラフを伝播し、最終的な出力に変換されるかを視覚的に表示します。
- **ライブ・パッチング**: 現実（DSP）と知覚（UI）の境界を維持したまま、動作中のオーディオをリアルタイムで再構成（Hot-Reload）することを可能にします。
- **真実の監視**: オーディオエンジンからのフィードバックを UI 上に投影し、開発者が「自分のコードが実際に何をしているか」を確信できるようにします。

## 2. 真実の色（Oklab の強制）

LimeStudio では、動的な UI 要素（メーター、アナライザー、モジュレーションリングなど）において線形 sRGB 補間を禁止しています。sRGB で色を混ぜようとすると、`cargo lime lint` がコードを拒否します。

### 2.1 問題： 「灰色の汚れ (Gray Smudge)」
人間は光を線形に知覚しません。ほとんどのブラウザや OS がデフォルトで使用している sRGB 色空間では、2 つの色の数学的な中間点が、元の色よりも「濁って」見えたり「暗く」見えたりすることがよくあります。

- **例**: 明るい青と明るい黄を sRGB で混ぜると、しばしば死んだような、あるいは「汚れた」ような灰色がかった緑になります。

### 2.2 解決策： Oklab
Oklab は **知覚的色空間** です。以下のように設計されています：
1.  **色相の一貫性**: 青から白への遷移が、意図せず紫を経由することはありません。
2.  **眼に線形な明度**: 「明るい」と「暗い」の 50% 中間点は、人間にとって実際に 50% の明るさに *感じられ* ます。

### 2.3 使い方
LimeStudio を使うために色彩科学者である必要はありません。変換を処理するヘルパーを提供しています。

**間違った方法 (sRGB - リンターにより拒否):**
```rust
// これは「視覚法違反」を引き起こします
let display_color = mix(color_a, color_b, t);
```

**正しい方法 (Oklab - 法):**
```rust
// これにより知覚的な一貫性が保証されます
let display_color = Color::mix_oklab(color_a, color_b, t);
```

### 2.4 なぜこれを強制するのか
オーディオツールにおいて、**UI の明瞭さは安全性** です。
クリッピングを示すためにメーターの色が緑から赤に変わる場合、その遷移は完全にスムーズで予測可能でなければなりません。sRGB の「濁った」中間点は、信号がクリップし始める正確な瞬間を隠してしまう可能性があります。LimeStudio では、「十分良い」デフォルトよりも **視覚的決定論** を選択します。

## 3. フォレンジックな決定論（Forensic Determinism）
 
LimeStudio のオーディオエンジンは、「なんとなく動く」ことよりも「再現可能であること」を重視します。これは、プロダクション環境でのトラブルシューティングや、学術的な音声合成において不可欠な要素です。
 
### 3.1 決定論的 PRNG (Deterministic Reality)
DSP 処理におけるランダム性は、外部の非決定論的な要素（システム時間など）に依存してはなりません。LimeStudio では、すべての擬似乱数生成（PRNG）に `DeterministicRng` (Xoshiro256++) を使用し、シード値によってビット単位で同一の出力を再現できることを保証しています。
 
### 3.2 テレメトリ・ブリッジ（The Pulse of Truth）
オーディオスレッドで何が起きているか（CPU 負荷、クリッピング、NaN の発生など）を知ることは、開発において極めて重要です。しかし、UI スレッドがオーディオスレッドを覗き見たり、ロックをかけたりすることは許されません。
 
LimeStudio は **Forensic Telemetry Bridge** を提供しています：
- **Lock-free 一方向通信**: オーディオスレッドから UI スレッドへ、`rtrb` (RingBuffer) を介して高速にイベントを流し込みます。
- **真実のストリーミング**: 静的な状態（Snapshot）とは別に、時間の経過とともに発生する「動的なイベント（Audit Events）」をリアルタイムで追跡できます。
- **安全第一**: もしテレメトリのバッファが一杯になった場合、システムはオーディオの継続を優先し、古いテレメトリを破棄します。

## 開発者向け要約

| 概念 | 従来の方法 | Lime の方法 | なぜ？ |
| :--- | :--- | :--- | :--- |
| **状態** | ウィジェットが値を所有 | カーネルが値を所有 | リアルタイム安全性と監査可能性 |
| **更新** | 直接的な変更 | Intent -> Snapshot | レースコンディションの排除 |
| **色** | sRGB `mix()` | Oklab `mix()` | 知覚的な誠実さ |
| **失敗** | ロジックバグ | コンパイルエラー (Lint) | 最終製品への確信 |
